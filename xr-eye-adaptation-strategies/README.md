# XR 亮度变化与人眼适应：设计策略规范

## 基于人眼明暗适应机制的 XR 环境亮度管理指南

---

## 1. 概述

本规范从资深 UI/UX 设计师视角出发，系统梳理人眼对亮度变化的适应机制，分析 XR 环境中亮度管理的核心挑战，并提炼出一套通用的 XR 设计策略。目标是帮助设计者在 XR 产品中合理控制亮度变化，减少视觉不适，提升长时间使用的舒适度和可用性。

### 1.1 适用范围

| 属性 | 说明 |
|---|---|
| 平台 | XR 头显（VR、AR、MR），通用，不绑定特定硬件 |
| 覆盖场景 | 场景切换、环境光变化、UI 明暗过渡、昼夜循环、passthrough 切换 |
| 核心关注 | 亮度（Luminance）、对比度（Contrast）、色温（Color Temperature）的时间维度管理 |
| 内容类型 | 系统 UI、应用内容、环境渲染、过渡动画、通知系统 |

### 1.3 核心设计哲学

**尊重人眼的时间维度。** 人眼不是瞬态响应系统，而是具有秒级到分钟级适应时间常数的生物传感器。XR 设计中一切亮度变化的节奏，都应匹配人眼的适应速度，而非追逐技术上的瞬时能力。

## 2. 人眼明暗适应的生理基础

### 2.1 三种视觉模式

人眼视网膜上的两类感光细胞——**视杆细胞（Rods）** 和 **视锥细胞（Cones）**——在不同亮度环境下交替主导视觉。

| 视觉模式 | 亮度范围（cd/m²） | 主导细胞 | 视觉特征 | 典型场景 |
|---|---|---|---|---|
| **暗视觉（Scotopic）** | 10⁻⁶ ~ 10⁻² | 视杆细胞 | 无色彩感知，仅灰度；峰值灵敏度约 498 nm（蓝绿色）；空间分辨力低 | 无月星夜、全暗房间 |
| **中间视觉（Mesopic）** | 10⁻² ~ 3.0 | 视杆 + 视锥 | 色彩部分可辨但不稳定；蓝色物体相对变亮（Purkinje 位移）；空间分辨力中等 | 黄昏、街灯照明、车内仪表 |
| **明视觉（Photopic）** | 3.0 ~ 10⁸ | 视锥细胞 | 完整色彩感知；最高空间分辨力；三类视锥（S/M/L）覆盖可见光谱 | 室内照明、日光 |

> **对 XR 的意义**：当前大多数 VR 头显的峰值亮度在 100~200 nits（cd/m²）范围内，使用户长期处于明视觉模式的中低段。场景渲染中若出现接近纯黑区域，用户视觉可能短暂滑入中间视觉，导致色彩感知不稳定。

### 2.2 适应机制与时间常数

明暗适应不是单一过程，而是多个生理机制协同运作的结果。

#### 2.2.1 明适应（Light Adaptation）：暗 → 亮

| 机制 | 响应速度 | 作用 |
|---|---|---|
| **瞳孔收缩** | 0.2 ~ 0.5 秒 | 瞳孔直径从约 8 mm 缩至约 2 mm，减少进光量约 16 倍；速度快但调节范围有限（仅约 1.2 log 单位） |
| **视锥细胞光化学适应** | 1 ~ 3 秒 | 视锥内的视色素（photopsin）漂白/再生平衡迅速调整，降低灵敏度以匹配新亮度 |
| **神经增益调整** | 50 ~ 200 毫秒 | 视网膜神经节细胞调整信号增益，压缩高亮度范围内的对比度信息 |
| **整体完成** | **约 1 ~ 5 秒** | 从全暗到明亮环境，大部分明适应在 5 秒内完成 |

#### 2.2.2 暗适应（Dark Adaptation）：亮 → 暗

| 机制 | 响应速度 | 作用 |
|---|---|---|
| **瞳孔扩张** | 1 ~ 5 秒 | 瞳孔直径从约 2 mm 扩至约 8 mm，增加进光量 |
| **视锥细胞暗适应** | 5 ~ 8 分钟 | 视锥细胞首先恢复灵敏度，约在 5~8 分钟达到最大暗适应灵敏度 |
| **锥-杆转换点（Kohlrausch Kink）** | 约 7 ~ 10 分钟 | 视杆细胞灵敏度超过视锥细胞，接管暗环境视觉 |
| **视杆细胞暗适应** | 20 ~ 40 分钟 | 视杆细胞内视紫红质（rhodopsin）完全再生，灵敏度持续提升 |
| **整体完成** | **约 30 ~ 40 分钟** | 完全暗适应需要较长时间，但主要可用灵敏度在前 10 分钟恢复 |

> **关键不对称性**：明适应（秒级）远快于暗适应（分钟级）。这意味着在 XR 中，**从暗到亮的过渡** 比 **从亮到暗的过渡** 给用户造成的即时不适更强烈（瞬间眩光），但恢复更快；而 **从亮到暗的过渡** 虽然即时不适较低，但视觉功能（清晰度、色彩）恢复缓慢。

### 2.3 瞳孔光反射与 XR 的特殊性

| 参数 | 自然环境 | XR 头显环境 |
|---|---|---|
| 光源距离 | 远距离，多方向 | 极近（屏幕距眼约 2~4 cm） |
| 亮度范围 | 0.001 ~ 100,000+ cd/m² | 通常 0.1 ~ 200 cd/m² |
| 光源面积占视野比例 | 通常较小（太阳约 0.5°） | 近乎 100%（屏幕覆盖全视野） |
| 周边环境光 | 持续可见 | VR 完全遮蔽；AR 部分透过 |
| 瞳孔反射触发 | 自然、渐进 | 完全由渲染内容控制 |

> **设计含义**：在 VR 中，渲染画面是瞳孔大小的**唯一决定者**。一帧画面的亮度突变就足以触发瞳孔反射。这赋予了 XR 设计者对用户瞳孔状态的间接但完全的控制权，也带来了相应的设计责任。

### 2.4 Purkinje 位移效应

当亮度从明视觉范围降至中间视觉范围时，视觉灵敏度峰值从绿黄色区域（约 555 nm，视锥主导）偏移至蓝绿色区域（约 507 nm，视杆主导）。

| 亮度状态 | 感知特征 | XR 设计影响 |
|---|---|---|
| 明视觉 | 红色鲜艳，蓝色相对暗淡 | 正常 UI 色彩方案可用 |
| 中间视觉过渡期 | 红色物体显著变暗，蓝色物体相对变亮 | 依赖红色的警告/强调信息可能失效 |
| 暗视觉 | 色彩几乎消失，仅剩灰度 | 色彩编码系统完全失效，需依赖亮度对比和形状 |

> **设计原则**：在 XR 暗场景中，**不应仅依赖色彩差异传递关键信息**。红色警告在低亮度下的可见性远低于设计者在明亮显示器上预览时的效果。应叠加亮度对比、形状差异或动态效果来确保信息可达。

### 2.5 眩光与不适眩光

眩光（Glare）是指视野中过高亮度或过大亮度对比导致的视觉功能下降或不适感。

| 眩光类型 | 机制 | XR 中的触发条件 |
|---|---|---|
| **失能眩光（Disability Glare）** | 眼内光散射形成光幕覆盖视网膜像，降低对比度感知 | 暗场景中突然出现高亮 UI 元素、白色弹窗、闪光特效 |
| **不适眩光（Discomfort Glare）** | 瞳孔在暗适应扩大状态下接收过量光通量，引发不适 | 暗场景快速切换至亮场景、VR 中从暗室突然进入户外环境 |
| **适应眩光（Adaptation Glare）** | 短暂暴露于高亮度后，视觉系统暂时灵敏度降低 | 闪光特效后的暂时"致盲"，从 passthrough（亮）切回虚拟暗环境后看不清 |

## 3. XR 显示系统的亮度特性

### 3.1 当前主流 XR 显示技术对比

| 参数 | LCD（大多数 VR） | OLED（部分 VR/AR） | Micro-OLED（高端 VR） | 光波导（AR 眼镜） |
|---|---|---|---|---|
| 典型峰值亮度 | 100 ~ 200 nits | 150 ~ 300 nits | 200 ~ 5,000 nits | 1,000 ~ 4,000 nits |
| 对比度 | 1,000:1 ~ 2,000:1 | 理论无穷（像素级关闭） | 理论无穷 | 受环境光影响极大 |
| 黑电平 | 较高（背光泄漏） | 极低（真黑） | 极低 | 不适用（叠加显示） |
| HDR 能力 | 有限 | 较好 | 优秀 | 受限 |
| 全局调光 | 可实现 | 逐像素控制 | 逐像素控制 | 可辅助全局调光 |

### 3.2 XR 显示亮度 vs. 真实世界亮度

```
真实世界亮度范围（cd/m²）：
├── 10⁻⁴  星光
├── 10⁻²  月光
├── 10⁰   烛光
├── 10²   室内照明              ◄── 当前大多数 VR 头显峰值（100~200 nits）
├── 10³   阴天户外
├── 10⁴   晴天户外
├── 10⁵   直视阳光
└── 约 14 个数量级（log 单位）

典型 VR 头显动态范围：
├── 最低可显示亮度：~0.01 ~ 0.1 nits
├── 典型峰值亮度：~100 ~ 200 nits
└── 约 3 ~ 4 个数量级
```

> **核心矛盾**：真实世界亮度跨越约 14 个数量级，人眼通过适应机制可应对约 10 个数量级的范围。但当前 VR 头显仅能呈现约 3~4 个数量级。这意味着 XR 必须通过**色调映射（Tone Mapping）**将真实世界的高动态范围压缩至显示设备的有限范围内，而这种压缩本身就会与人眼的自然适应预期产生冲突。

### 3.3 AR 叠加显示的特殊挑战

AR 设备（如 HoloLens、Magic Leap）使用**叠加式（Additive）** 显示——虚拟像素的光叠加在真实环境光之上。

| 挑战 | 说明 | 设计影响 |
|---|---|---|
| **无法显示黑色** | 像素"关闭"时仍可见真实环境，虚拟元素无法遮挡 | 深色 UI 背景在亮环境下变透明；需用暗色 + 不透明背板 |
| **环境光竞争** | 明亮户外环境下虚拟内容被"冲淡" | 虚拟元素亮度必须足以与环境竞争，但过亮会导致眼疲劳 |
| **亮度叠加** | 每个亮虚拟像素都在向用户眼中增加光通量 | 大面积白色 UI 在 AR 中相当于手电筒直射，应避免 |
| **不可控环境变化** | 用户移动时环境亮度实时变化（室内→室外） | 系统必须实时感知环境光并调整虚拟内容亮度 |

## 4. XR 中的典型亮度变化场景与不适来源

### 4.1 场景分类

本章将场景分为**系统级**（操作系统/平台层面触发）和**应用级**（应用内容触发）两大类。

#### 系统级场景

| # | 场景类型 | 亮度变化特征 | 典型触发 | 不适风险等级 |
|---|---|---|---|---|
| S1 | **Passthrough 切换** | VR 暗环境 ↔ 真实世界（通常更亮） | 双击头显外壳、系统安全边界触发、Guardian/边界提示 | ⚠️⚠️⚠️ **高** |
| S2 | **应用间跳转** | 两个应用的平均亮度可能差异极大 | 从暗色游戏切到亮色浏览器、从沉浸式体验返回系统主页 | ⚠️⚠️⚠️ **高** |
| S3 | **系统 UI 弹出** | 暗背景上叠加亮色系统面板 | 系统菜单、通知中心、虚拟键盘、权限弹窗 | ⚠️⚠️ **中** |
| S4 | **开机/唤醒序列** | 从全黑到系统启动画面 | 用户戴上头显、从休眠唤醒 | ⚠️⚠️ **中** |
| S5 | **AR 环境光变化** | 用户移动导致环境光实时变化 | 从室内走到户外、转向窗户、灯光开关 | ⚠️⚠️ **中** |

#### 应用级场景

| # | 场景类型 | 亮度变化特征 | 典型触发 | 不适风险等级 |
|---|---|---|---|---|
| S6 | **场景硬切（Hard Cut）** | 瞬时全画面亮度跳变 | 瞬移传送、关卡切换、剧情转场 | ⚠️⚠️⚠️ **高** |
| S7 | **环境光渐变（Gradual Ambient）** | 秒到分钟级的全局亮度渐变 | 日出日落模拟、天气变化、洞穴明暗交替 | ⚠️ **低** |
| S8 | **局部高亮（Local Highlight）** | 画面局部出现高亮度区域 | 爆炸特效、手电筒光锥、魔法闪光、UI 弹窗 | ⚠️⚠️ **中** |
| S9 | **应用内 UI 叠加** | 暗背景上叠加亮色应用 UI 面板 | 背包界面、地图覆盖、设置面板 | ⚠️⚠️ **中** |
| S10 | **闪烁/频闪（Flicker）** | 高频亮度变化（>3 Hz） | 特效、状态指示、受伤/损坏提示、频闪灯 | ⚠️⚠️⚠️ **高**（光敏癫痫风险） |

### 4.2 不适产生的生理路径分析

```
亮度突变
  │
  ├──→ 瞳孔反射延迟（0.2~0.5s）──→ 暂时过量/不足进光 ──→ 即时不适/眩光
  │
  ├──→ 视网膜适应滞后（秒~分钟级）──→ 对比度感知失调 ──→ 细节丢失/看不清
  │
  ├──→ 视锥-视杆切换不稳（中间视觉）──→ 色彩感知波动 ──→ 色彩不确定感
  │
  └──→ 重复性触发 ──→ 持续性瞳孔震荡（hippus 加剧）──→ 眼肌疲劳 ──→ 头痛
```

### 4.3 高风险场景详解

#### S1 Passthrough 切换

**设计情境**：用户在沉浸式 VR 体验（中低亮度）中触发 passthrough 查看真实环境（通常更亮）。

| 特殊性 | 说明 |
|---|---|
| 亮度落差大 | VR 虚拟场景平均亮度可能仅 20~50 nits，而 passthrough 呈现的真实环境可达 500+ nits |
| 色温跳变 | 虚拟场景色温由设计者控制（常用暖色调），真实环境色温取决于实际照明 |
| 景深跳变 | 从虚拟定焦距内容切换到真实多焦距环境，调节-辐辏冲突叠加 |
| 用户预期 | 用户主动触发时有心理准备，系统自动触发（如安全边界）时无预期 |

#### S2 应用间跳转

**设计情境**：用户从一个平均亮度很低的沉浸式游戏（如恐怖游戏的暗色场景），通过系统菜单跳转到一个亮色应用（如白色背景的浏览器或协作工具）。

| 阶段 | 时间 | 用户体验 |
|---|---|---|
| 系统菜单弹出 | 0 ~ 0.3 秒 | 暗场景中出现亮色系统 UI，首次亮度冲击 |
| 选择目标应用 | 用户操作时间 | 瞳孔已部分适应系统 UI 亮度 |
| 目标应用加载 | 0 ~ 0.5 秒 | 如果目标应用比系统 UI 更亮，产生二次冲击 |

> **设计机会**：系统菜单本身可以作为一个"中间亮度站"——其亮度设置为当前应用与系统平均亮度的中间值，为瞳孔提供过渡缓冲。

#### S3 系统 UI 弹出

**设计情境**：用户在暗色场景中收到一条系统通知，一个白底弹窗出现在视野中央。

| 问题 | 生理机制 | 感受 |
|---|---|---|
| 暗适应状态下瞳孔已扩大 | 瞳孔直径可能在 5~7 mm | 白色弹窗的光通量直接冲击扩大的瞳孔 |
| 弹窗占据中央视野 | 中央凹直接接收高亮度刺激 | 瞬间眩光感，周围暗场景暂时"看不见" |
| 弹窗消失后 | 瞳孔需重新扩张适应暗环境 | 适应眩光——几秒内暗场景细节丢失 |

#### S4 开机/唤醒序列

**设计情境**：用户戴上头显，从全黑状态进入系统启动画面。

| 阶段 | 最佳实践 | 反面案例 |
|---|---|---|
| 全黑 → 启动画面 | 从极低亮度（如 5% 峰值）渐亮至正常亮度，持续 2~3 秒 | 直接显示全亮 logo 画面 |
| 启动画面 → 主页 | logo 亮度与主页亮度保持一致或平滑过渡 | logo 用纯白背景，主页用深色主题 |

#### S6 场景硬切（应用内）

**设计情境**：用户在暗色洞穴环境中游戏，一次瞬移传送到了阳光明媚的室外。

| 阶段 | 时间 | 用户体验 |
|---|---|---|
| 切换瞬间 | 0 ~ 0.2 秒 | 白光冲击，本能眯眼或闭眼 |
| 瞳孔收缩中 | 0.2 ~ 1 秒 | 画面过亮但正在改善 |
| 视锥适应中 | 1 ~ 5 秒 | 色彩过饱和感，高光处细节丢失 |
| 基本适应完成 | 5 ~ 10 秒 | 视觉恢复正常 |

## 4.4 从生理原理到设计策略：推导总览

本节将第 2 章的生理学发现与第 5 章的设计策略做显式对应，展示每条策略"从何而来"。

```
生理学原理                                    设计策略
─────────────────────────────────────────────────────────────────────

明适应需 1~5 秒，暗适应需数分钟              ──→  D1 渐变过渡
  人眼不能瞬时切换亮度灵敏度                        所有亮度变化的时长 ≥ 适应时间

瞳孔反射有 0.2~0.5 秒延迟                    ──→  D2 预适应提示
  突变亮度时瞳孔来不及反应                          提前微调亮度方向，预启动瞳孔调节

VR 中渲染画面是瞳孔的唯一光源                 ──→  D3 亮度感知管理
  画面亮度直接决定瞳孔状态                          控制画面整体亮度变化速率和峰值

人眼对对比度的敏感性 > 对绝对亮度的敏感性     ──→  D4 对比度保护
  低亮度≠不可用，低对比度=不可读                    亮度调节时对比度不可低于阈值

Purkinje 位移使暗环境中红色失效               ──→  D5 暗场景色彩策略
  视杆细胞主导时色彩感知退化且偏移                  暗场景配色需适配视杆细胞灵敏度曲线

光敏性癫痫由高频亮度变化触发                  ──→  D6 闪烁安全控制
  XR 全视野覆盖使风险高于传统屏幕                  严格限制闪烁频率和面积

人眼持续适应环境光（自然本能）                ──→  D7 环境感知自适应
  AR 虚拟内容叠加在变化的真实环境上                虚拟内容亮度需跟随环境光协同变化

适应能力因年龄、健康状态而异                  ──→  D8 用户控制与个性化
  40+ 岁暗适应时间增加 1.5~2 倍                    提供个性化参数，不做"一刀切"
```

## 5. 通用 XR 亮度适应设计策略

> **策略总纲**：所有策略围绕一个核心原则——**将任何亮度变化的速率控制在人眼适应速率之内，并在不可避免的快速变化前给予用户生理和心理上的缓冲。**

### 5.1 策略一览

| 编号 | 策略名称 | 适用场景 | 核心要点 | 生理学依据 |
|---|---|---|---|---|
| D1 | 渐变过渡（Gradual Transition） | S1, S2, S4, S6 | 用时间换舒适；所有亮度变化均采用渐变曲线 | §2.2 适应时间常数 |
| D2 | 预适应提示（Pre-Adaptation Cue） | S1, S2, S6 | 在大幅亮度变化前，提前微调画面亮度方向 | §2.3 瞳孔反射延迟 |
| D3 | 亮度感知管理（Luminance Perception Management） | S3, S8, S9 | 控制画面整体亮度的变化速率和峰值分布 | §2.3 VR中瞳孔控制权 |
| D4 | 对比度保护（Contrast Preservation） | 全场景 | 在调整亮度的同时维持足够的对比度层次 | §2.2 + §2.5 |
| D5 | 暗场景色彩策略（Dark Scene Chromatics） | S6 暗段, S7 暗段 | 在低亮度场景中使用适应人眼暗视觉特性的色彩方案 | §2.1 + §2.4 Purkinje位移 |
| D6 | 闪烁安全控制（Flash Safety Control） | S10 | 严格限制闪烁频率、面积和累积暴露 | §2.5 眩光机制 |
| D7 | 环境感知自适应（Ambient-Aware Adaptation） | S5, AR 全场景 | AR 设备根据环境光实时调节虚拟内容亮度和对比度 | §2.2 + §3.3 |
| D8 | 用户控制与个性化（User Control） | 全场景 | 提供亮度偏好设置，尊重个体差异 | §2.2 个体差异 |

### 5.2 D1：渐变过渡（Gradual Transition）

> **生理学依据 → §2.2**：明适应涉及瞳孔收缩（0.2~ 0.5s）和视锥光化学调整（1~ 3s），整体约需 1~ 5 秒。暗适应更慢——视锥阶段需 5~ 8 分钟，视杆阶段需 20~ 40 分钟。**任何快于这些时间常数的亮度变化都会造成适应滞后，表现为眩光或暂时看不清。**

#### 核心原理

人眼明适应约需 1~ 5 秒，暗适应的初始阶段约需 5~ 10 秒可恢复基本可用视力。所有全画面亮度变化应在这一时间范围内或更长时间内完成。

#### 参数建议

| 过渡方向 | 建议最短持续时间 | 推荐曲线 | 说明 |
|---|---|---|---|
| **暗 → 亮** | 1.5 ~ 3.0 秒 | Ease-in（慢启快停） | 前半段慢速升亮让瞳孔先行收缩，后半段可加速；避免亮度陡升段出现在过渡前 30% |
| **亮 → 暗** | 2.0 ~ 4.0 秒 | Ease-out（快启慢停） | 快速降低峰值亮度以减少总光照暴露，然后缓慢降至目标暗度让视锥细胞适应 |
| **同亮度级切换** | 0.5 ~ 1.0 秒 | Ease-in-out | 即使目标亮度相近，画面内容变化也可能引起局部对比度跳变 |

#### 实现方式

| 方法 | 适用技术栈 | 描述 |
|---|---|---|
| **全局 Post-Processing Fade(ease-in, ease-out)** | Unity / Unreal | 在后处理阶段叠加亮度渐变滤镜（如乘以 0→1 或 1→0 的 alpha 曲线） |
| **Tone Mapping 参数渐变(ease-in-out)** | 自研渲染管线 | 渐变 Tone Mapping 曲线的曝光参数（exposure），而非线性叠加遮罩 |
| **场景交叉溶解（Cross-Dissolve）** | 通用 | 两个场景并行渲染，旧场景淡出的同时新场景淡入，保持亮度平滑 |
| **中间过渡帧** | 通用 | 在两个场景之间插入中间亮度状态的过渡帧序列 |

#### 渐变曲线形状参考

```
亮度
 ▲
 │      ╭──────── 目标亮度
 │    ╱
 │  ╱               ← Ease-in（暗→亮推荐）：初始斜率小，末段斜率大
 │╱                    瞳孔在前段低斜率期完成收缩
 ├────────────────── 时间 →

 ▲
 │─────╮
 │      ╲
 │        ╲          ← Ease-out（亮→暗推荐）：初始斜率大，末段斜率小
 │          ╲───── 目标亮度
 ├────────────────── 时间 →
```

### 5.3 D2：预适应提示（Pre-Adaptation Cue）

> **生理学依据 → §2.3**：瞳孔光反射存在 0.2~0.5 秒的固有延迟，且完整收缩/扩张需要更长时间。**如果在亮度突变之前提前"预热"瞳孔调节，真正切换时的亮度冲击将大幅减小。** 这类似于人从暗室走向门口时，门缝透进的光线让眼睛提前开始适应。

#### 核心原理

在大幅亮度变化前 0.5~2 秒内，将当前画面亮度向目标方向微调 10%~20%，启动瞳孔的提前调节，减少实际切换时的亮度冲击。

#### 实施方法

| 场景 | 预适应方式 | 时间线 |
|---|---|---|
| **暗→亮场景切换** | 在切换前 1~2 秒，逐渐提高当前暗场景的全局亮度（如提亮 15%~20%），或在画面边缘引入目标场景的光晕（ambient glow） | T-2s: 开始提亮 → T-0.5s: 达到预适应亮度 → T-0: 执行切换渐变 |
| **亮→暗场景切换** | 在切换前 1~2 秒，逐渐压低当前亮场景的高光区域，或引入暗色渐晕（vignette） | T-2s: 开始压暗 → T-0.5s: 达到预适应亮度 → T-0: 执行切换渐变 |
| **Passthrough 激活** | 在 passthrough 画面完全显示前，先显示一帧模糊、降低对比度的 passthrough 预览 | T-0.3s: 模糊低对比度预览 → T-0: 渐变至完整 passthrough |
| **Passthrough 关闭** | 在返回 VR 前，先将 passthrough 画面做一次渐暗处理 | T-0.5s: passthrough 开始渐暗 → T-0: 渐变至 VR 场景 |

#### 设计效果

```
传统硬切：  暗场景 ████████│████████ 亮场景     ← 瞬间跳变，瞳孔来不及反应
                         ↑ 不适峰值

预适应+渐变：暗场景 ██████▓▓▒▒░░░░████ 亮场景     ← 提前缓冲 + 平滑过渡
                       ↑ 预适应开始  ↑ 切换点       ← 无明显不适峰值
```

### 5.4 D3：亮度感知管理（Luminance Perception Management）

> **生理学依据 → §2.3**：在 VST 中，渲染画面是瞳孔大小的唯一决定者。画面中一个大面积高亮区域的突然出现，等同于在用户眼前突然开灯。**设计师对画面亮度分布的每一个决定，都直接影响用户的瞳孔状态和视觉舒适度。**

#### 核心原理

控制画面整体亮度的变化速率和高亮区域的分布，避免渲染内容对瞳孔造成突发性刺激。这不是一个纯技术问题——设计师在规划场景光照、UI 配色和特效方案时，就已经在做"亮度管理"。

#### 设计师需要关注的四个维度

| 维度 | 设计师的判断方式 | 风险信号 | 建议 |
|---|---|---|---|
| **画面整体亮度** | 场景整体给人"亮"还是"暗"的感觉 | 上一个场景很暗，这个场景突然很亮 | 与前一场景的亮度差控制在 30% 以内，超出则需渐变过渡 |
| **高亮区域面积** | 画面中白色/高亮部分占多大比例 | 超过画面 60% 的区域是高亮的 | 压缩高亮面积，或降低高亮区域的绝对亮度 |
| **暗区域面积** | 画面中接近黑色的区域占多大比例 | 超过画面 80% 都是暗的（瞳孔会充分扩张） | 暗场景中避免突然出现高亮元素；若必要，先渐亮再显示 |
| **亮暗交界锐度** | 亮区和暗区之间的边界是否生硬 | 纯白面板直接放在纯黑背景上 | 在亮暗交界处增加柔和渐变（如阴影、光晕、模糊边缘） |

#### 常见设计场景的亮度管理

| 设计场景 | 问题 | 改进方案 |
|---|---|---|
| 暗场景中弹出白色通知 | 瞳孔扩大状态下遭遇高亮冲击 | 通知使用深色背景 + 浅色文字；或先显示半透明暗色遮罩再渐亮显示通知 |
| 游戏中洞穴内火把/爆炸 | 暗环境中的局部高亮造成适应眩光 | 控制火焰/爆炸的峰值亮度不超过画面平均亮度的 5 倍；使用扩散光晕而非硬边光源 |
| 从暗色菜单返回明亮场景 | 菜单期间瞳孔已适应低亮度 | 菜单关闭时先显示 0.5 秒的渐亮过渡，而非瞬间恢复场景亮度 |
| AR 中大面积白色 UI 面板 | 叠加显示导致光通量过大 | 使用深色/半透明背景；白色元素仅用于文字和图标，不做大面积填充 |

#### 研究支撑

2025 年的 Temporal Brightness Management for Immersive Content 研究表明，基于内容感知的实时亮度管理可将眼睛疲劳指标降低约 45%，同时不显著影响画质评分。

### 5.5 D4：对比度保护（Contrast Preservation）

> **生理学依据 → §2.2 + §2.5**：人眼的对比度灵敏度函数（CSF）表明，人眼检测亮度差异的能力远优于判断绝对亮度的能力。在 Weber 定律描述的范围内，人眼能检测到约 1%~2% 的亮度差异。这意味着——**降低整体亮度通常不会严重影响可用性，但降低对比度会立即让信息变得不可读。** 此外，失能眩光（§2.5）通过光散射降低视网膜像的对比度，是许多亮度相关不适的根源。

#### 核心原理

调整亮度时必须同步保护对比度。人眼对对比度的敏感程度远高于对绝对亮度的敏感程度——在较低亮度下仍可正常阅读，但一旦对比度下降至阈值以下，信息立即变得不可读。

#### 最低对比度要求

| 内容类型 | 最低亮度对比度（L*单位） | WCAG 等效参考 | 说明 |
|---|---|---|---|
| **正文文字** | ≥ 40 L* | ≥ 4.5:1 | 正文文字在 XR 中已受限于分辨率，对比度不可再打折 |
| **大标题** | ≥ 25 L* | ≥ 3:1 | 大号文字容许较低对比度 |
| **关键操作按钮** | ≥ 40 L* | ≥ 4.5:1 | 主要交互元素不可因亮度调节而丢失边界 |
| **装饰性元素** | ≥ 15 L* | — | 不承载关键信息的元素可适当放宽 |
| **危险/警告信息** | ≥ 50 L* | ≥ 7:1 | 安全类信息应维持最高对比度 |

> **L*单位:** CIE Lab色彩空间中用来表示明度的数值坐标。L = 0, 纯黑色；L = 100, 纯白色；L = 50, 中间灰度。
> **WCAG**: web无障碍指南。

#### 亮度调节中的对比度保护策略

| 策略 | 方法 | 效果 |
|---|---|---|
| **比例缩放（Proportional Scaling）** | 整体亮度乘以同一系数 | 保持相对对比度不变，但暗部可能被压到不可见 |
| **抬高黑位（Black Level Lift）** | 降低亮度时同时抬高最低黑位 | 避免暗部细节完全消失，但会降低暗端对比度 |
| **自适应对比度增强** | 在亮度压缩后，对关键 UI 元素的对比度做局部增强 | 保障 UI 可读性，但可能使 UI 与背景的融合感降低 |
| **语义优先（Semantic Priority）** | 标记关键 UI 元素，亮度调节时保证其对比度不低于阈值 | 最灵活但需要渲染管线配合 |

### 5.6 D5：暗场景色彩策略（Dark Scene Chromatics）

> **生理学依据 → §2.1 + §2.4**：当亮度降至中间视觉范围，视杆细胞开始参与视觉。视杆细胞的峰值灵敏度在 507 nm（蓝绿区），对 620~700 nm（红色区）几乎无响应。Purkinje 位移使红色物体在低亮度下显著变暗，蓝色物体相对变亮。同时，视杆细胞的空间分辨力远低于视锥细胞。**这意味着你在 Figma 的明亮显示器上设计的暗色 UI，用户在头显暗场景中看到的效果可能完全不同——红色警告可能几乎不可见，细小文字可能无法辨认。**

#### 核心原理

在低亮度场景中，利用人眼暗视觉特性优化色彩方案，而非简单套用明视觉下设计的配色。

#### 设计规则

| 规则 | 说明 | 原理 |
|---|---|---|
| **R1：暗场景中避免依赖红色传递关键信息** | 红色在低亮度下最先失去亮度感知（视杆细胞对长波不敏感） | Purkinje 位移——视杆细胞峰值灵敏度在 507 nm，远离红色的 620~700 nm |
| **R2：暗场景中偏好蓝绿色系作为功能色** | 蓝绿色在低亮度下相对最亮，可维持辨识度 | 视杆细胞峰值灵敏度（507 nm）恰好位于蓝绿区 |
| **R3：暗场景 UI 使用高亮度对比而非色彩对比** | 亮度差异在低光下比色彩差异更可靠 | 暗视觉下色彩感知退化，亮度对比是唯一可靠的视觉分离手段 |
| **R4：暗场景中增大 UI 元素尺寸** | 视杆细胞空间分辨力低于视锥细胞，暗视觉下精细细节难以辨认 | 暗适应后中央凹功能减弱（中央凹以视锥为主），周边视力相对提升 |
| **R5：深色 UI 模式不等于"适配暗视觉"** | 深色模式降低平均亮度但通常保持明视觉色彩方案，不应视为暗视觉优化 | 深色模式的白色文字仍处于明视觉亮度范围，色彩感知机制不变 |

#### 暗场景推荐调色板

| 用途 | 推荐色域 | 避免色域 | 理由 |
|---|---|---|---|
| 主要信息文字 | 低饱和蓝白 / 浅青色 | 纯白（过亮）/ 纯红（Purkinje 失效） | 兼顾可读性和眼睛舒适度 |
| 警告/错误 | 琥珀色（amber）+ 形状标记 | 纯红色（暗中不可见） | 琥珀色波长较短，辅以三角形等形状标记 |
| 成功/确认 | 青绿色 | 纯绿色（暗中饱和度下降） | 青绿色接近视杆细胞峰值灵敏度 |
| 背景 | 深灰（#1A1A1A ~ #2D2D2D） | 纯黑（#000000） | 纯黑在 OLED 上与任何内容形成极端对比，引起视觉不适 |
| 分割线/边框 | 中灰（#555555 ~ #777777） | 高饱和彩色线条 | 暗场景中彩色线条的色彩信息不可靠 |

### 5.7 D6：闪烁安全控制（Flash Safety Control）

> **生理学依据 → §2.5**：高频亮度变化可触发光敏性癫痫发作。传统屏幕的安全标准（如 WCAG 2.3.1 的 3 次/秒限制）基于屏幕仅占视野一部分的假设。**XR 头显覆盖用户几乎全部视野，等效的刺激面积远大于手机或电视屏幕**——同等参数的闪烁在 XR 中的风险等级显著高于传统媒介。此外，反复的亮度突变会导致瞳孔在收缩和扩张之间持续震荡（hippus 加剧），引发眼肌疲劳和头痛。

#### 核心原理

符合光敏性癫痫安全标准（基于 ITU-R BT.1702 / Harding 测试 / WCAG 2.3.1），在 XR 中由于显示覆盖全视野，风险比传统屏幕更高。

#### 安全阈值

| 参数 | 安全限制 | XR 特殊考虑 |
|---|---|---|
| **一般闪烁** | 每秒不超过 3 次全画面闪烁 | XR 中"全画面"即全视野，比电视屏幕覆盖更多视网膜面积 |
| **红色闪烁** | 严格避免交替红/黑或红/白闪烁 | 红色闪烁是最高风险类型；即使在暗场景中也应避免 |
| **闪烁面积** | 不超过视野面积的 25%（传统标准） | XR 中建议更保守——不超过中央 10° 视野面积（对应中央凹区域） |
| **累积闪烁暴露** | 连续闪烁持续时间不超过 2 秒 | 即使频率在安全范围内，持续闪烁仍会引发不适 |

#### 设计替代方案

| 避免 | 替代 |
|---|---|
| 画面闪白 / 闪黑 | 柔和的亮度脉动（pulse），幅度控制在 20% 以内 |
| 频闪警告灯效果 | 持续性发光 + 缓慢亮度呼吸（0.5~1 Hz） |
| 爆炸白屏闪光 | 从爆炸中心向外扩散的光晕（radial bloom），持续 0.5 秒以上 |
| 伤害红屏闪烁 | 边缘渐晕（vignette）红色叠加，缓入缓出，0.3 秒以上 |

### 5.8 D7：环境感知自适应（Ambient-Aware Adaptation）——AR 专用

> **生理学依据 → §2.2 + §3.3**：在自然环境中，人眼持续且无意识地适应环境光变化。AR 设备在真实环境上叠加虚拟内容，如果虚拟内容的亮度不跟随环境光变化——用户走到户外时虚拟 UI 被"冲淡"看不清，走入暗室时虚拟 UI 又变得刺眼——**这种虚拟与真实光照环境的不一致，会打破人眼自然的适应节奏，造成额外的视觉负担。**

#### 核心原理

AR 设备必须实时感知环境光并相应调整虚拟内容的亮度、对比度和色温，以保证在各种照明条件下虚拟内容的可见性、舒适度和与真实环境的视觉融合感。

#### 调节维度

| 维度 | 感知方式 | 调节策略 |
|---|---|---|
| **环境亮度** | 光传感器 / passthrough 图像分析 | 明亮环境提升虚拟内容亮度；暗环境降低虚拟内容亮度 |
| **环境色温** | 色温传感器 / 白平衡分析 | 虚拟内容色温向环境色温靠拢，维持视觉融合感 |
| **环境对比度** | 场景分析 | 高对比环境中增强虚拟内容边缘对比度 |
| **用户位置/朝向变化** | IMU + 视觉 SLAM | 预判亮度变化趋势（如用户转向窗户），提前调整 |

#### 调节速度

| 环境变化类型 | 建议响应时间 | 理由 |
|---|---|---|
| 缓慢环境变化（云层遮日） | 2 ~ 5 秒渐变 | 匹配自然感知，避免虚拟内容"呼吸感" |
| 快速环境变化（走入室内） | 0.5 ~ 1.5 秒渐变 | 比人眼适应略快，维持可读性 |
| 极端环境变化（黑暗中开灯） | 0.3 ~ 0.8 秒渐变 | 快速响应保护眼睛，但不应瞬变 |

### 5.9 D8：用户控制与个性化（User Control）

该模块的结论没有验证，待以后涉及后继续补充。

> **生理学依据 → §2.2 个体差异**：暗适应速度随年龄显著下降——40 岁以上用户的暗适应时间可达年轻用户的 1.5~ 2 倍，瞳孔最大扩张直径也随年龄减小（60 岁约 5~ 6 mm vs. 20 岁约 7~ 8 mm），导致暗场景下可用光通量更低。白内障早期患者的眼内光散射增加，眩光敏感度显著升高。**一套固定参数无法同时满足 20 岁和 60 岁用户，更无法照顾有视觉健康问题的用户。**

#### 核心原理

人眼适应能力存在显著个体差异（年龄、眼部健康、药物影响等），单一的系统策略无法满足所有用户。

#### 必要的用户控制项

| 控制项 | 默认值建议 | 范围 | 说明 |
|---|---|---|---|
| **全局亮度偏好** | 系统中值（50%） | 0% ~ 100% | 基础亮度偏移，叠加在所有自动调节之上 |
| **过渡速度偏好** | "标准"（对应推荐渐变时间） | "快速" / "标准" / "舒适" | 控制所有过渡动画的时间缩放系数 |
| **夜间模式** | 自动（基于时间或环境光） | 开 / 关 / 自动 | 降低蓝光、降低整体亮度、切换暗色 UI |
| **闪烁敏感度** | "标准" | "标准" / "敏感"（更严格过滤）| 敏感模式下进一步降低闪烁阈值 |
| **对比度增强** | 关 | 开 / 关 | 为低视力用户提供额外的对比度增强 |

#### 个体差异参考数据

| 因素 | 对适应能力的影响 | 设计考虑 |
|---|---|---|
| **年龄增长（40+）** | 暗适应时间增加至 1.5~2 倍；瞳孔最大扩张减小；透光率下降 | 老年用户需要更长的过渡时间和更高的基础亮度 |
| **近视/矫正** | 通常无显著影响，但高度近视可能影响周边视觉 | 非主要因素 |
| **白内障早期** | 光散射增加，眩光敏感度显著升高 | 需要更激进的眩光保护和更低的峰值亮度 |
| **散瞳药物** | 瞳孔无法正常收缩，明适应机制受损 | 极端情况，系统应提供"散瞳模式"大幅降低亮度 |

### 5.10 设计案例：常见场景的改进前后对比

以下案例展示如何将上述策略应用于实际 XR 设计场景。

#### 案例 1：VR 游戏关卡传送（暗→亮）

**情境**：玩家从地下城（平均亮度 15 nits）传送至沙漠地表（平均亮度 160 nits），亮度差约 10 倍。

| | 改进前 | 改进后 |
|---|---|---|
| **过渡方式** | 屏幕瞬间切换到沙漠场景 | 分三阶段过渡 |
| **阶段 1** | — | **预适应**（D2）：传送触发前 1.5 秒，地下城出口处渐渐出现金色天光，整体亮度提升 15% |
| **阶段 2** | — | **渐变**（D1）：传送触发时，用 2 秒的 Ease-in 曲线从地下城亮度升至沙漠亮度 |
| **阶段 3** | — | **着陆**：沙漠场景初始曝光略低于最终值，在 1 秒内升至目标曝光 |
| **用户体验** | 白光冲击，眯眼 3~5 秒 | 感觉"走出洞穴，眼睛渐渐适应阳光"——自然且沉浸 |
| **运用策略** | — | D1 + D2 |

#### 案例 2：系统通知在暗场景中弹出

**情境**：用户在暗色 VR 影院中观影（平均亮度 8 nits），收到一条系统通知。

| | 改进前 | 改进后 |
|---|---|---|
| **通知样式** | 白底黑字弹窗（亮度约 180 nits） | 深灰底（#2D2D2D）+ 浅青色文字，亮度约 25 nits |
| **出现方式** | 立即弹出 | 0.3 秒淡入，先显示半透明暗色遮罩 |
| **位置** | 视野中央 | 视野下方 15°（近周边视觉区域），避免刺激中央凹 |
| **消失方式** | 点击后立即消失 | 0.3 秒淡出 |
| **用户体验** | 被闪了一下，几秒内看不清电影画面 | 注意到通知但不受打扰，视觉无中断 |
| **运用策略** | — | D1（渐变出入） + D3（控制亮度峰值） + D5（暗场景配色） |

#### 案例 3：Passthrough 切换

**情境**：用户在 VR 冥想应用中（深蓝色空间，平均亮度 10 nits），双击头显切换到 passthrough（室内光照约 300 lux，passthrough 画面约 80 nits）。

| | 改进前 | 改进后 |
|---|---|---|
| **切换方式** | 0.1 秒内完成切换 | 分预适应 + 渐变两步 |
| **预适应** | 无 | 切换前 0.3 秒：显示一帧低对比度、模糊化的 passthrough 预览（D2） |
| **渐变** | 无 | 0.8 秒内从模糊预览渐变至完整 passthrough（D1） |
| **返回 VR** | 瞬间回到暗色冥想空间 | passthrough 先渐暗 0.5 秒，再用 1 秒渐变至 VR 场景 |
| **用户体验** | 去程被亮光闪到，回程觉得突然变黑看不清 | 双向过渡自然，不打断冥想节奏 |
| **运用策略** | — | D1 + D2 |

#### 案例 4：暗场景 UI 配色

**情境**：为一款太空探索 VR 游戏设计 HUD（背景是深空，平均亮度 5 nits）。

| UI 元素 | 改进前 | 改进后 | 依据 |
|---|---|---|---|
| 血量条 | 红色（#FF0000） | 琥珀色（#FFB347）+ 条形轮廓 | D5-R1：红色在暗视觉中失效 |
| 警告图标 | 红色三角 | 琥珀色三角 + 脉动动画（0.5Hz） | D5-R1 + D5-R3：叠加形状和动态编码 |
| 导航文字 | 14px 白色细体 | 18px 浅青色（#A8E6CF）粗体 | D5-R4：暗场景增大尺寸 + D5-R2：蓝绿色系 |
| 背景面板 | 纯黑（#000000） | 深灰（#1A1A2E） | D5 调色板：纯黑与内容形成极端对比 |
| 能量指示 | 蓝绿渐变条 | 保留（与暗视觉灵敏度匹配） | D5-R2：蓝绿色在低亮度下可见性最佳 |

#### 案例 5：AR 工业巡检应用

**情境**：工人佩戴 AR 眼镜在工厂中移动，环境光从 500 lux（车间）变化到 50,000 lux（室外装卸区）再到 20 lux（仓库）。

| 环境 | 虚拟内容调整 | 策略 |
|---|---|---|
| 车间（500 lux） | 标准亮度，白色文字 + 深色半透明背板 | D7 基准状态 |
| 走向室外途中 | 检测到亮度升高趋势，提前 2 秒开始增强虚拟内容亮度和对比度（D7） | D7 预判调节 |
| 室外（50,000 lux） | 虚拟内容亮度拉满，不透明黑色背板，文字加粗加描边 | D7 + D4 |
| 走入仓库途中 | 检测到亮度下降趋势，1.5 秒内渐降虚拟内容亮度（D7） | D7 + D1 |
| 仓库（20 lux） | 虚拟内容降至低亮度，使用蓝绿色系功能色，增大文字尺寸 | D7 + D5 |

## 7. 测试与验证方法

### 7.1 测试环境矩阵

| 环境条件 | 描述 | 测试关注点 |
|---|---|---|
| **暗室（< 1 lux）** | 完全黑暗的室内环境 | 暗场景色彩策略效果、瞳孔扩张后的亮部冲击 |
| **室内照明（100~500 lux）** | 标准办公/家居环境 | 常规使用体验、UI 对比度 |
| **明亮室内（500~1000 lux）** | 靠窗位置、明亮灯光 | AR 虚拟内容可见性 |
| **户外阴影（1,000~5,000 lux）** | 阴天或阴影区域 | AR 在中等户外光下的表现 |
| **户外直射光（10,000~100,000 lux）** | 晴天户外 | AR 在强光下的极端测试 |

### 7.2 测试场景清单

| 测试编号 | 测试场景 | 验证要点 | 通过标准 |
|---|---|---|---|
| T1 | 暗场景 → 亮场景硬切 | 过渡平滑度，无眩光不适 | 受试者不适评分 ≤ 2/10 |
| T2 | 亮场景 → 暗场景切换 | 暗场景信息可读性恢复时间 | 关键 UI 在 3 秒内可读 |
| T3 | VR ↔ Passthrough 切换 | 过渡自然度，无明显亮度冲击 | 受试者主观评价"自然"或"较自然" |
| T4 | 暗场景中弹出亮色 UI | 瞳孔冲击、UI 周围眩光光晕 | 无可感知的光晕/光幕 |
| T5 | 长时间低亮度使用（30min+）| 色彩辨识准确性、眼疲劳 | 色彩辨识正确率 > 90%，不适评分 ≤ 3/10 |
| T6 | 爆炸/闪光特效 | 闪烁安全性、主观不适 | 符合 Harding 测试，不适评分 ≤ 2/10 |
| T7 | 快速环境光变化（AR）| 虚拟内容可见性和可读性 | 文字始终可读，过渡无闪烁 |
| T8 | 老年用户（55+）专项 | 适应速度、亮度偏好、对比度需求 | 所有 UI 在用户偏好亮度下可读 |

## 8. 设计检查清单（Design Checklist）

在产品设计评审中，使用以下清单逐项确认亮度适应相关设计的合规性：

### 8.1 过渡与动画

- [ ] 所有全画面亮度变化是否采用渐变过渡（D1）？
- [ ] 渐变持续时间是否 ≥ 推荐最短时间？
- [ ] 大幅亮度变化前是否有预适应提示（D2）？
- [ ] Passthrough 切换是否有专用过渡方案（D2）？

### 8.2 对比度与可读性

- [ ] 正文文字对比度是否 ≥ 40 L*（D4）？
- [ ] 关键操作按钮对比度是否 ≥ 40 L*（D4）？
- [ ] 警告/安全信息对比度是否 ≥ 50 L*（D4）？
- [ ] 亮度调节后对比度是否仍满足最低要求（D4）？

### 8.3 暗场景

- [ ] 暗场景中是否避免了纯红色作为唯一信息载体（D5-R1）？
- [ ] 暗场景 UI 是否使用了增大尺寸的元素（D5-R4）？
- [ ] 暗场景背景是否避免了纯黑（#000000）（D5 调色板）？
- [ ] 暗场景中关键信息是否有亮度对比 + 形状双编码（D5-R3）？

### 8.4 闪烁安全

- [ ] 是否确认无超过 3 次/秒的全画面闪烁（D6）？
- [ ] 是否避免了红色闪烁（D6）？
- [ ] 闪光特效是否替换为扩散光晕等安全方案（D6）？
- [ ] 连续闪烁是否限制在 2 秒以内（D6）？

### 8.5 AR 专项

- [ ] 是否有环境光感知和自适应调节（D7）？
- [ ] 虚拟内容亮度是否在各种环境光下均可见（D7）？
- [ ] 是否避免了大面积白色/亮色 UI（D7 + HoloLens 指南）？
- [ ] 环境光变化时的调节是否为渐变而非跳变（D7）？

### 8.6 用户控制

- [ ] 是否提供全局亮度调节选项（D8）？
- [ ] 是否提供过渡速度偏好设置（D8）？
- [ ] 是否支持闪烁敏感模式（D8）？
- [ ] 是否考虑了老年用户的适应能力差异（D8）？
